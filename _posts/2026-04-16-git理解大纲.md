## 大纲

---

### 1. Git 的对象模型

**1.1 三种核心数据类型**
- 数据对象（blob）：文件内容的快照
- 树对象（tree）：目录结构，指向 blob 和子 tree
- 提交对象（commit）：指向 tree + parent + 元数据

**1.2 存储方式：内容寻址**
- `.git/objects/` 的目录结构
- SHA-1 的计算方式（header + content）
- loose object 与 packfile

---

### 2. commit 对象的内部结构

**2.1 commit 的组成字段**
- tree / parent / author / committer / message

**2.2 parent 指针与 SHA 的关系**
- 为什么改一个字符整个 SHA 都变
- merge commit 的双 parent 结构

---

### 3. branch 的本质

- branch 只是一个指向 commit SHA 的文本文件
- HEAD 是指向 branch 的指针（detached HEAD とは）
- branch の移動コスト O(1)

---

### 4. git merge 与三路合并算法

**4.1 fast-forward merge**
- 不产生新 commit，只移动指针

**4.2 三路合并（three-way merge）**
- 为什么需要三路：two-way 的局限
- Base / Ours / Theirs 的确定方式（merge-base）
- 冲突的产生条件

**4.3 merge commit 的结构**
- 两个 parent 指针

---

### 5. git log 与 DAG

- commit 历史有向无环图（DAG）
- `git log` 的遍历算法（DFS + 拓扑排序）

---

### 6. git reflog 的本质

- reflog 是什么（本地操作日志，与 DAG 无関）
- 保存内容：每次 HEAD 移动的记录
- 与 `git log` 的本质区别
- 数据丢失恢复的原理

---

### 7. git reset 与 git revert

- reset：移动 HEAD/branch 指针（改写历史）
  - `--soft` / `--hard` 的区别
- revert：创建新 commit 抵消变更（保留历史）
- 为什么公共分支禁止 reset

---

### 8. git rebase 与 git cherry-pick

**8.1 cherry-pick**
- 本质：对单个 commit 的patch（差分）取出 并生成新 SHA
- git cherry-pick 中的三路合并与git merge的区别

**8.2 rebase**
- 本质：逐个 cherry-pick + 移动 branch 指针
- `--onto` 的工作原理
- 为什么公共分支禁止 rebase

**8.3 rebase vs merge 的使用场景**

---

### 9.git branch rule

```
Case 1 限制条件：
  - 未 push 到公共分支，OR
  - 已 push 但公共分支处于 code freeze（大家都知道）
  → 两种情况下都可以在个人分支或公共分支上执行 stash → pull → stash pop
```

```
Case 2（main，chore/A ∥ feature/B）：
chore/A -> main
main --> feature/B 
 L 有冲突: X 因为main
 L 无冲突: OK

git checkout feature/B 
git pull origin/main （featureB -> main 但是没有merge提交 只显示解决冲突与拉新代码）

git merge origin/main

  有冲突：
    先 release 的分支 → 切出 feature/A-B-merge
    PR: 后 release → feature/A-B-merge
    冲突修正落在「后 release 分支」上
  无冲突：
    PR: 先 release → feature/B  直接合
```

```
Point 3

      feature -> main 
             L c1 
before: main -> feature
     (auto-merge/fast-forward) PR merged closed

  创建 PR 前检查反向 PR（总之如果你不知道开没开起auto-merge，还需要观察其他PR）
  → 存在则先 close，再创建新 PR
  → 而非仅警告（这是与之前 GitHub Actions 方案的关键区别）
```
