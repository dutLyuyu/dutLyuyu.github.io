## 1. Git のオブジェクトモデル

### 1.1 三種類のコアデータ型

Git は「バージョン管理システム」である前に、**内容アドレス型ファイルシステム**である。
すべてのデータは三種類のオブジェクトとして `.git/objects/` に保存される。

#### blob（データオブジェクト）

ファイルの「内容」だけを保存する。ファイル名もパーミッションも持たない。

```
$ echo "hello" | git hash-object --stdin
ce013625030ba8dba906f756967f9e9ca394464a

$ git cat-file -p ce013625030ba8dba906f756967f9e9ca394464a
hello
```

同じ内容のファイルは何個あっても **SHA が同じ = 同一の blob オブジェクト**。
Git がストレージ効率に優れる理由のひとつ。

#### tree（ツリーオブジェクト）

ディレクトリ構造を表す。blob と子 tree への参照を持つ。

```
$ git cat-file -p HEAD^{tree}
100644 blob ce013625...    README.md
100644 blob 7b57bd29...    package.json
040000 tree 9a3d82f1...    src/
```

各エントリは `モード / 型 / SHA / 名前` の形式。
**ファイル名はここで初めて登場する**（blob は名前を持たない）。

#### commit（コミットオブジェクト）

tree + parent + メタデータを持つ。

```
$ git cat-file -p HEAD
tree   9a3d82f1...
parent a1b2c3d4...
author  Alice <alice@example.com> 1700000000 +0900
committer Alice <alice@example.com> 1700000000 +0900

feat: add login feature
```

**tree はそのコミット時点のプロジェクト全体のスナップショット**を指す。
差分ではなく、常に全体像を保存している点が重要。

---

### 1.2 保存方式：内容アドレッシング

オブジェクトの SHA は「ヘッダ + コンテンツ」に対して計算される。

```
SHA-1( "blob 6\0hello\n" )
       ↑型  ↑サイズ ↑内容
```

計算後、先頭 2 文字をディレクトリ名、残り 38 文字をファイル名として保存する。

```
.git/objects/
  ce/
    013625030ba8dba906f756967f9e9ca394464a  ← blob
  a1/
    b2c3d4...                               ← commit
```

**内容が同じなら SHA は常に同じ**。これが Git の整合性保証の根拠。
内容が 1 バイトでも変われば SHA は完全に別の値になる（なぜ後述）。

#### loose object と packfile

コミットが増えると個別ファイル（loose object）が大量に発生する。
Git は定期的に `git gc` で packfile にまとめ、差分圧縮して効率化する。

```
.git/objects/pack/
  pack-abc123.idx    ← SHA → offset のインデックス
  pack-abc123.pack   ← 実データ（差分圧縮済み）
```

## 2. commit オブジェクトの内部構造

### 2.1 commit の構成フィールド

| フィールド | 内容 |
|---|---|
| `tree` | そのコミット時点のルートディレクトリの SHA |
| `parent` | 直前のコミットの SHA（merge commit は 2 つ） |
| `author` | 変更を作成した人と日時 |
| `committer` | commit を記録した人と日時（rebase で変わる） |
| `message` | コミットメッセージ |

### 2.2 parent ポインタと SHA の連鎖

**なぜ 1 バイト変えると SHA が全く変わるのか：**

```
commit C の SHA = SHA-1("commit ...\ntree T\nparent B\n...")
commit B の SHA = SHA-1("commit ...\ntree T'\nparent A\n...")
```

B の内容を変えると B の SHA が変わる。
すると C の `parent` フィールドが参照していた B の SHA が変わるため、C の SHA も変わる。
以降すべての子孫 commit の SHA が連鎖的に変わる。

**この連鎖が「歴史の改ざん検知」を実現している。**

### 2.3 merge commit の double parent 構造

```
$ git cat-file -p <merge-commit>
tree   fa3b91...
parent 7246dc3...    ← first parent  (ours: head branch)
parent 94bfc57...    ← second parent (theirs: merged branch)
```

merge commit は通常の commit と区別されるフォーマットを持たない。
**parent が 2 本あるだけ**。これだけで 2 つのブランチの履歴がひとつに繋がる。

## 3. branch の本質

### 3.1 branch は commit SHA へのポインタファイル

```
$ cat .git/refs/heads/main
a1b2c3d4e5f6...   ← 40 文字の SHA テキストだけ

$ cat .git/refs/heads/feature
7246dc3a13d7...
```

branch は SHA を書いた**テキストファイル 1 枚**に過ぎない。
「ブランチを作る」= このファイルを 1 つ作るだけ。コストは O(1)。

### 3.2 HEAD：現在地を指すポインタ

```
$ cat .git/HEAD
ref: refs/heads/feature    ← 通常: branch への参照

$ git checkout a1b2c3d4    ← detached HEAD 状態
$ cat .git/HEAD
a1b2c3d4e5f6...            ← 直接 commit SHA を指す
```

commit 時、Git は HEAD → branch → 旧 SHA を読み、
新しい commit を作り、branch ファイルの SHA を新しいものに書き換える。

```
commit 前: main → a1b2c3
commit 後: main → f9e8d7  (f9e8d7.parent = a1b2c3)
```

### 3.3 branch の移動コスト

どんなに長い歴史を持つブランチでも、
「ブランチを切る」「切り替える」「削除する」はすべてファイルの読み書きだけ。
コミット数に関係なく O(1)。

## 4. git merge と三路合並算法

### 4.1 fast-forward merge

**成立条件：** merge 先（base）が merge 元（head）の ancestor である場合。

```
main:    A ── B
               \
feature:        C ── D  (HEAD)

git checkout main
git merge feature
→ main は D に移動するだけ。新しい commit は生まれない。

main:    A ── B ── C ── D  (main が D を指す)
```

**何も起きない**のが fast-forward の本質。
新 commit を生成しないため、`merge commit = feature HEAD の SHA` が成立する。
これが本事故の直接的な原因だった。

強制的に merge commit を生成したい場合は `--no-ff` を使う：

```bash
git merge --no-ff feature -m "merge feature into main"
```

### 4.2 三路合并（three-way merge）が必要な理由

**two-way merge の限界：**

```
main:    "hello world"
feature: "hello git"

→ 変更したのはどちらか分からない。両方が変えた？片方だけ？
```

2 つのスナップショットだけでは「どちらが変更したか」が判断できない。

**three-way merge：**

```
Base (共通祖先): "hello"
Ours  (main):   "hello world"   ← main が "world" を追加
Theirs (feature): "hello git"   ← feature が "git" を追加

→ 両方が独立して変更 → 冲突
→ 一方だけ変更なら自動マージ可能
```

### 4.3 merge-base（共通祖先）の特定

Git は `git merge-base` コマンドで自動的に共通祖先を特定する。

```bash
git merge-base main feature
# → 両ブランチが分岐した地点の commit SHA を返す
```

```
      common (Base)
      /            \
main: A ── B ── C (Theirs)
feature: P ── Q (Ours)

Base  = common
Ours  = feature HEAD (Q)
Theirs = main HEAD (C)
```

**冲突の発生条件：**

```
Base と Ours の diff: 変更あり
Base と Theirs の diff: 同じ箇所に変更あり
→ 冲突

Base と Ours の diff: 変更あり
Base と Theirs の diff: 同じ箇所に変更なし
→ 自動マージ（Ours の変更を採用）
```

### 4.4 merge commit と ours/theirs の方向性

```
git checkout A
git merge B
→ merge commit の parents = { A(first/ours), B(second/theirs) }
→ merge commit は A のブランチに落ちる
→ B のブランチポインタは動かない
```

**A → B と B → A は全く異なる操作。**

| | A(head) → B(base) | B(head) → A(base) |
|---|---|---|
| merge commit が落ちる場所 | A ブランチ | B ブランチ |
| ours | A | B |
| theirs | B | A |
| fast-forward の判定 | B が A の ancestor か | A が B の ancestor か |

## 5. git log と DAG

### 5.1 commit 履歴は「リスト」ではなく DAG

**DAG（有向非巡回グラフ）：** エッジが一方向で、サイクルを持たないグラフ。

```
A ── B ── C ── D          # 直線の場合はリストに見える
              ↑
              merge
             / \
        E ── F   G ── H   # merge が入ると DAG になる
```

merge commit が 2 本の parent を持つため、
履歴は必然的にグラフ構造になる。

### 5.2 git log の遍历算法

`git log` は HEAD から始まる **深さ優先探索（DFS）+ トポロジカルソート**。

```
1. HEAD を queue に入れる
2. queue から commit を取り出し、出力
3. その commit の全 parent を queue に追加
4. queue が空になるまで繰り返す
（すでに訪問済みの commit はスキップ）
```

merge commit の 2 本の parent は両方たどられるため、
**merge 後は両ブランチの全 commit が `git log` で見える**。
これが「冲突解決後、feature の log に main の M1/M2 が現れる」理由。

### 5.3 --first-parent と --graph

```bash
# merge commit の first parent（ours）だけをたどる
git log --first-parent

# グラフ表示
git log --oneline --graph --all
```

`--first-parent` は merge した側のメインラインだけを追跡し、
取り込まれた側の履歴を非表示にする。リリース履歴の確認に有用。

## 6. git reflog の本質

### 6.1 reflog とは何か

`git log` が **commit オブジェクトの DAG を辿る**のに対し、
`git reflog` は **HEAD（およびブランチ）がいつどこを指していたか**のローカル操作ログ。

```
$ git reflog
a1b2c3d HEAD@{0}: commit: feat: add login
f9e8d7c HEAD@{1}: checkout: moving from feature to main
7246dc3 HEAD@{2}: reset: moving to HEAD~1
```

### 6.2 git log との本質的な違い

| | git log | git reflog |
|---|---|---|
| 何を見るか | commit DAG の永続的な履歴 | HEAD の移動履歴（ローカルのみ） |
| 保存場所 | `.git/objects/` | `.git/logs/HEAD` |
| リモートに push | される | されない |
| 保持期間 | 永続 | デフォルト 90 日 |

### 6.3 データ復旧の原理

`git reset --hard HEAD~1` でコミットが「消えた」ように見えても、
commit オブジェクト自体は `.git/objects/` に残っている。
reflog でその SHA を確認し、`git checkout <SHA>` で復元できる。

```bash
# reset で消えた commit を復元
git reflog
# → abc1234 HEAD@{1}: commit: 消えた commit

git checkout abc1234
# または
git reset --hard abc1234
```

オブジェクトは `git gc` が走るまで（通常 30 日）削除されない。

## 7. git reset と git revert

### 7.1 git reset：ポインタを移動する（歴史の書き換え）

reset は branch ポインタを指定した commit に移動させる操作。

```
before:  A ── B ── C  (HEAD, main)
git reset --hard B
after:   A ── B  (HEAD, main)
                 C はオブジェクトとして残るが、辿る手段がない
```

**三つのモードの違い：**

| モード | ブランチポインタ | インデックス | ワーキングツリー |
|---|---|---|---|
| `--soft` | 移動 | そのまま | そのまま |
| `--mixed`（default） | 移動 | リセット | そのまま |
| `--hard` | 移動 | リセット | リセット |

### 7.2 git revert：新しい commit で打ち消す（履歴を保持）

```
before:  A ── B ── C  (HEAD)
git revert C
after:   A ── B ── C ── C'  (C' は C の変更を逆にした commit)
```

C は履歴に残る。`C'` は `C の変更を打ち消す差分` を適用した新しい commit。

### 7.3 公共ブランチで reset が禁止される理由

```
remote: A ── B ── C  (main)
自分:   git reset --hard B → A ── B

git push --force
remote: A ── B  ← C が消える

他の人: A ── B ── C ── D  (C を基にした作業が無効になる)
```

reset + force push は **他人の branch の親 commit を破壊する**。
公共ブランチでは必ず revert を使う。

## 8. git rebase と git cherry-pick

### 8.1 cherry-pick：1 つの commit を別の場所に複製する

```
git cherry-pick <commit-sha>
```

指定した commit の差分を取り出し、**現在の HEAD に新しい commit として適用する**。

```
before:
  main:    A ── B ── C
  feature: A ── D ── E

git checkout main
git cherry-pick E

after:
  main:    A ── B ── C ── E'  (E' は E と同内容だが SHA は別)
  feature: A ── D ── E        (E は残る)
```

**E と E' は内容が同じでも SHA が異なる別の commit。**
後で feature を main に merge すると同じ変更が 2 回適用される可能性がある（重複 commit）。

#### cherry-pick の三路合并

cherry-pick も内部では三路合并アルゴリズムを使う。
ただし三点の取り方が `git merge` と異なる。

```
git merge B の三点:
  Base   = merge-base(A, B)  ← 両ブランチの共通祖先
  Ours   = A HEAD
  Theirs = B HEAD（B の全変更）

git cherry-pick C の三点:
  Base   = C^                ← C の親 commit（1つ前）
  Ours   = 現在の HEAD
  Theirs = C
```

**merge の Base は「分岐点」、cherry-pick の Base は「その commit の直前」。**
これが冲突範囲の差を生む本質的な違い。

#### 三路合并の判定ルール

各行について次のルールで判定する：

```
Ours のみ変更    → Ours を採用（冲突なし）
Theirs のみ変更  → Theirs を採用（冲突なし）
両方とも変更     → 冲突
両方とも変更なし  → そのまま（冲突なし）
```

#### 具体例で見る冲突の発生条件

以下の状態を例に取る：

```
common の f1.txt:          main の変更:            feature の変更:
  line1: common              M1: line1 を変更         C1: line2 を変更
  line2: common              M2: line3 を変更
  line3: common

main:    common ── M1 ── M2
feature: common ── C1
```

**CASE A: git merge main（Base = common）**

```
        Base(common)    Ours(C1)             Theirs(M2)          判定
line1:  "common"      → "common"(変更なし)   "changed in M1"   → Theirs 採用のはずが…
line2:  "common"      → "changed in C1"     "common"(変更なし) → Ours 採用のはずが…
line3:  "common"      → "common"(変更なし)   "changed in M2"   → Theirs 採用のはずが…
```

→ **冲突発生。** Base が古い（common）ため M1・M2 の変更が一括で Theirs に入り、
複数行の変更が重なって Git の行マッチングが崩れる。

**CASE B: cherry-pick M1（Base = common）**

M1 は line1 だけを変更。C1 は line2 だけを変更。触る行が別。

```
        Base(common)    Ours(C1)              Theirs(M1)          判定
line1:  "common"      → "common"(変更なし)    "changed in M1"   → Theirs 採用 ✅
line2:  "common"      → "changed in C1"      "common"(変更なし) → Ours 採用  ✅
line3:  "common"      → "common"(変更なし)    "common"(変更なし) → そのまま   ✅
```

→ **冲突なし。** cherry-pick は M1 が変えた差分だけを対象にするため
C1 との衝突が起きない。

**CASE C: cherry-pick M2（Base = M1）**

M2 の直前は M1。しかし feature の C1 は M1 を知らない（common から切っている）。

```
        Base(M1)             Ours(C1)             Theirs(M2)          判定
line1:  "changed in M1"    → "common"            "changed in M1"   → Ours と Theirs が食い違う → 冲突 ❌
line2:  "common"           → "changed in C1"     "common"(変更なし) → Ours 採用 ✅
line3:  "common"           → "common"(変更なし)   "changed in M2"   → Theirs 採用 ✅
```

→ **冲突発生。** C1 は M1 を含まないため、
cherry-pick から見ると「Ours が line1 を "changed in M1" から "common" に戻した」
と解釈される。Theirs(M2) はそのままのため食い違いが生じる。

#### cherry-pick で冲突が回避できる条件

```
条件①: ピックする commit が触った行と Ours が触った行が重ならない

条件②: Ours の中に Base（C^）の内容が含まれている
        = Ours が Base の変更を「知っている」

②が崩れる典型ケース:
  feature が main の途中 commit を知らない状態で
  その後の commit を cherry-pick する
  → Base の内容が Ours に存在しない → 冲突
```

「cherry-pick なら冲突しない」ではなく、
**「cherry-pick は無関係な変更を冲突判定から除外できる」** が正確な理解。

### 8.2 rebase：branch の付け根を付け替える

```
git rebase <newbase>
```

現在のブランチの各 commit を newbase の上に順番に cherry-pick し直す。

```
before:
  main:    A ── B ── C
  feature: A ── D ── E

git checkout feature
git rebase main

after:
  main:    A ── B ── C
  feature: A ── B ── C ── D' ── E'  (D, E は新 SHA で作り直される)
  (旧 D, E は参照がなくなる)
```

D → D', E → E' は **内容は同じだが SHA が変わる新しい commit**。

### 8.3 git rebase --onto

通常の rebase は「ブランチ全体を newbase に移動」するが、
`--onto` は「どこからどこまでを移動するか」を指定できる。

```bash
git rebase --onto <newbase> <exclude-from> <branch>
```

```
例: personal を feature から main に移植する

before:
  main:    A ── B ── C
  feature: A ── D ── E
  personal: A ── D ── E ── P1 ── P2

git rebase --onto main $(git merge-base feature personal) personal

after:
  main:    A ── B ── C
  feature: A ── D ── E
  personal: A ── B ── C ── P1' ── P2'  (D, E は含まれない)
```

`merge-base feature personal` が「D, E をスキップして P1 から移植する」ための anchor になる。

### 8.4 公共ブランチで rebase が禁止される理由

rebase は既存の commit を**すべて別の SHA で作り直す**（cherry-pick の連続）。
これは履歴の書き換えであり、他の人がその旧 SHA を親として持っている場合、
その全 commit が孤立する。

```
公共 feature（他の人が clone 済み）:
  remote: A ── D ── E

自分が feature で rebase:
  remote: A ── D' ── E'  (force push)

他の人:
  local:  A ── D ── E ── P1 ── P2
  D, E が消えた → P1, P2 の parent が宙吊りになる
```

### 8.5 rebase vs merge の選択基準

| 状況 | 推奨 |
|---|---|
| 個人ブランチを最新 main に追従させる | rebase |
| 公共ブランチに main の変更を取り込む | merge |
| 特定の 1 commit だけを別ブランチに持ち込む | cherry-pick（重複に注意） |
| 公共ブランチで操作全般 | rebase / cherry-pick 禁止、merge のみ |

## まとめ：Git の本質を一枚で

```
┌─────────────────────────────────────────────────────┐
│  .git/objects/                                       │
│                                                      │
│  blob: ファイル内容（名前なし）                         │
│  tree: ディレクトリ構造（blob + tree への参照）          │
│  commit: tree + parent(s) + メタデータ                 │
│                                                      │
│  すべてのオブジェクトは SHA-1 で識別される               │
│  内容が同じなら SHA は常に同じ                          │
└─────────────────────────────────────────────────────┘
         ↓ commit の parent チェーンで繋がる
┌─────────────────────────────────────────────────────┐
│  commit の DAG（有向非巡回グラフ）                      │
```
